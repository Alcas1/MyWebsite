WHAT IS REST?
Representational State Transfer
  • Restful programming is coding on the weekend on the couch
  • A means of data exchange on the web
  • A set of constraints that define an architectural style
  • Used to develop web APIs
Constraints
  1. Client - Server
  2. Stateless
  3. Cacheable
  4. Uniform Interface
  5. Layered System
  6. Code on Demand(Optional)
Client - Server
  • Clients retrieve data through requests
  • Servers store and serve data
  • Servers don't care about state
  • Code portability
  • Scalability
  • Responsibilities seperated
Stateless
  • RE (State) T is stateless
  • State information is contained within the request
  • Information about state is passed back through response
  • Servers can handle any state
  • Scalability (Handle requests more efficiently)
Cacheable
  • Some requests may be cached
  • Eliminates some interactions
  • Improves efficiency (handling more requests)
  • Improves latency (skip request)
  • Improves scalability (each server can do more)
Uniform Interfaces
  • Provides a standard for interaction
  • Abstraction of the implementation from the interface
  • Gives consistency across various applications
Layered System
  • Applications can have layered logic
  • Layers cannot see past other layers
  • Client can't tell whether its an intermediary server or an end server
Code on Demand(Optional)
  • Executable code transferred to client externally
  • Code sent by a server
  • Extends functionality
  • Not required for REST
RESOURCES & CRUD
Resources
  • Most common data model is JSON
  • JSON is key, value pairs
  • Resources should be nouns (not verbs!)
  • After defining resources, configure usages
  • Example resources: ticket, user, course
Resource Example
Ticket Resource

Request:
GET /tickets/2
Response:
{
 id : 2,
 subject: "Why is it called cat?"
 priority : "low",
 state : "open"
}
CRUD
  • Create, Read, Update, Delete
  • HTTP verbs match CRUD
  • CREATE => POST
  • READ => GET
  • UPDATE => PUT
  • DELETE => DELETE
Tickets Example
  • /tickets <- endpoint
  • GET /tickets
  • GET /tickets/2
  • POST /tickets
  • PUT /tickets/2
  • DELETE /tickets/2

BEST PRACTICES
Versioning
  • Always version always
  • Offering of new api while still supporting old ones
  • Helps prevent invalid requests
  • API version can be included in the url
  • GET /api/v2/tickets
Result Filtering
  • Allows selection of specific content
  • API base URL should be as lean as possible
  • Easily implementable
  • Reduces latency
  • GET /ticket?priority=severe
Documentation
  • These apply mainly to public APIs but should be applied internally as well
  • API's are only good as their documentation
  • Easy to find and access
  • Most devs read docs before integrating
  • Docs should include complete request/response cycles
Key Points
  • REST is a set of constraints that define an architectural style
  • REST APIs are made to scale
  • Resources should be nouns
  • HTTP verbs allow for CRUD
  • Always version your API
  • Good documentation!
